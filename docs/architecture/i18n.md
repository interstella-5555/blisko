# Internationalization (i18n)

> v1 --- created 2026-05-20 (BLI-280).

PL is the source locale. UA is the only secondary locale today. Mobile uses [Lingui](https://lingui.dev) v6 with the React/macros API; backend templates (push, email) will get a small JSON-based helper in [BLI-281](https://linear.app/blisko/issue/BLI-281) / [BLI-282](https://linear.app/blisko/issue/BLI-282) — out of scope here.

Locale storage and per-device sync semantics live in [user-profiles.md](./user-profiles.md) (the `profiles.locale` column) and [mobile-architecture.md](./mobile-architecture.md#localestore) (the `localeStore`). This doc covers the runtime / build pipeline only.

## Stack

| Layer | Package | Version |
|-------|---------|---------|
| Runtime | `@lingui/core`, `@lingui/react` | v6 |
| Build macros | `@lingui/babel-plugin-lingui-macro` | v6 |
| Bundler integration | `@lingui/metro-transformer/expo` | v6 |
| CLI (extract + compile) | `@lingui/cli` | v6 |
| PO format | `@lingui/format-po` | v6 |
| PO parsing (dev scripts) | `gettext-parser` | v9 |

Why Lingui (vs i18next): Claude Code is the primary editor. Lingui keeps the source PL text inline at the call site (`<Trans>To nie przypadek</Trans>`) instead of behind opaque keys, so the agent sees the user-visible string directly without an extra Read into a JSON file. PO is a standard translator format if a human linguist ever joins.

## How a string flows through the system

1. Dev writes `<Trans>To nie przypadek</Trans>` or `t\`Spróbuj ponownie\`` in a component.
2. The Lingui babel macro transforms it at bundle time into `i18n._({id: "...", message: "..."})`. The `id` is a stable hash; the `message` is the original PL text.
3. `bun run mobile:i18n:extract` runs the Lingui CLI, which AST-scans `app/` and `src/` and produces `src/locales/pl/messages.po` (and a parallel skeleton in `uk/messages.po`). Each entry contains the `msgid` (original PL) and `msgstr` (translation — empty for UA on first extract).
4. `bun run mobile:i18n:translate` reads the PL PO + UA PO, sends every msgid missing a `msgstr` in UA to OpenAI `gpt-4o-mini` in batches of 30, and writes the translations back into `uk/messages.po`. **Idempotent** — existing UA strings are never overwritten, so re-running after an extract only fills the newly-added entries.
5. At app startup, `src/lib/i18n.ts` imports both PO files. Metro's `@lingui/metro-transformer/expo` turns each `.po` file into an ES module exporting `{ messages: Record<string, string> }`. `i18n.load(locale, messages)` registers both catalogs, then `i18n.activate(initialLocale)` switches to whichever locale `localeStore` says is active.
6. `app/_layout.tsx` subscribes to `useLocaleStore`; when the user taps the PL/UA toggle, the listener calls `i18n.activate(state.locale)`, which the `<I18nProvider>` propagates as a re-render down the tree.

## Macros — which one when

| You want to translate | Import | Pattern |
|-----------------------|--------|---------|
| JSX text inside a component | `import { Trans } from "@lingui/react/macro";` | `<Trans>Tekst</Trans>` |
| String prop / attribute inside a component (e.g. `placeholder`, `title`, toast msg) | `import { useLingui } from "@lingui/react/macro";` then `const { t } = useLingui();` | `t\`Tekst\`` |
| String outside the React tree (Zustand store actions, util fns, module-level helpers) | `import { t } from "@lingui/core/macro";` | `t\`Tekst\`` (the core-macro `t` must be called **inside** a function, not at module load) |

Both `t` macros support template-literal interpolation: `t\`Witaj ${name}\`` extracts as `msgid "Witaj {0}"` with the placeholder preserved.

For pluralization use `<Plural>` (React) or the `plural` macro (core). Ukrainian has three plural forms (one / few / many) — Lingui handles the ICU MessageFormat rules out of the box.

## Build-time pipeline

| Command | Purpose | When |
|---------|---------|------|
| `bun run mobile:i18n:extract` | Scan code for macros, update PL PO + sync the UA skeleton | After every meaningful UI change |
| `bun run mobile:i18n:translate` | Fill missing UA msgstrs via OpenAI | After extract, before commit |
| `bun run mobile:i18n:check` | CI gate: fail if any PL msgid has an empty UA msgstr | On every PR (see `.github/workflows/ci.yml` → `i18n` job) |

`lingui compile` is **not** in our workflow: `@lingui/metro-transformer/expo` (see `apps/mobile/metro.config.js`) transforms PO files inline during Metro bundling. Skipping the compile step keeps the dev iteration cycle one command shorter and avoids a generated `.js` artifact that would otherwise need to live alongside the PO source.

## OpenAI translator details

- Model: `gpt-4o-mini`. Cheap, fast, good for short UI copy. Upgrade to `gpt-4o` only if quality regressions appear on idiomatic phrasing.
- Batch size: 30 msgids per API call. The script splits any larger run into chunks; each batch is one chat completion with `response_format: { type: "json_object" }`.
- System prompt enforces: preserve `{placeholders}`, preserve `\n`, informal Ukrainian (`ти`-form), keep "Blisko"/"BLISKO" as-is, keep emails/URLs untouched.
- The script reads `OPENAI_API_KEY` from `process.env`. Export it from `apps/api/.env.production` before running (`apps/api/.env` has an empty value locally):

  ```bash
  export OPENAI_API_KEY=$(grep '^OPENAI_API_KEY=' /Users/karol/code/blisko/apps/api/.env.production | cut -d= -f2-)
  bun run mobile:i18n:translate
  ```

- The script writes to UA PO **in place** with `gettext-parser`. Always review the diff (`git diff apps/mobile/src/locales/uk/messages.po`) before committing — the LLM occasionally drops a translation for very long strings (the script logs a warning and you can re-run; idempotency makes retries safe).

## CI

The `i18n` job in `.github/workflows/ci.yml` runs `bun run mobile:i18n:check` on every PR. It fails the build if any PL msgid lacks a UA translation. Combined with the `extract → translate → commit` workflow, this prevents shipping UI in PL-only state. Biome stays our linter; we don't add `@lingui/eslint-plugin` (no AST-aware "raw string" detection), but the CI gate + PR review catch the same class of bug.

## Edge cases / gotchas

- **Lingui v6 + React Native + Reanimated 4.** Reanimated 4 dropped its own babel plugin in favour of codegen, so before BLI-280 the project had no `babel.config.js`. Adding `babel.config.js` for Lingui requires reintroducing `babel-preset-expo` explicitly — without it, Expo's auto-resolved preset stops applying.
- **Macros at module load.** `t` from `@lingui/core/macro` resolves against the currently-active locale at call time. If you call it at module load (top-level constant), it captures the locale active at module load, which is whatever activated first — usually wrong. Always wrap top-level catalogs in a function (see `src/lib/rateLimitMessages.ts` for the pattern: switch/case inside `getRateLimitMessage()` instead of a module-level `Record`).
- **Cold-start flicker.** `localeStore` persists to SecureStore via Zustand's `persist` middleware — hydration is async. `i18n.ts` reads `useLocaleStore.getState().locale` synchronously at module load, so the very first render uses whichever default the store starts with (`"pl"`). When hydration completes the listener in `_layout.tsx` calls `i18n.activate(...)` and the tree re-renders. For UA users this means a one-frame PL flash on cold start. Acceptable for now; if we ever need to eliminate it, the AppGate splash can extend its hold past the `localeStore` hydration as well as the auth restore.
- **First-install seeding lives in `localeStore`, not in `i18n.ts`.** The store's `create()` initial value computes the OS-derived locale at module load via `detectLocaleFromLanguageCode()`; persist hydration then either replaces it with a saved value or leaves it in place to be persisted on the next `setLocale` call. See [mobile-architecture.md#localestore](./mobile-architecture.md#localestore).
- **OS-shipped strings stay native.** iOS / Android system dialogs (permission prompts, share sheet, etc.) follow the OS locale, not ours. We can't translate them; we can only word our own surrounding copy clearly.

## Backend templates

The API runs on Bun without a babel build step, so Lingui's macros (used in mobile) would require pre-compilation just for ~15 strings — overkill. `apps/api/src/services/i18n.ts` exposes a single function:

```ts
t(key: BackendTranslationKey, locale: string | null | undefined, params?: Record<string, string | number>): string
```

The pattern: **centralized typed catalog** at the top of `i18n.ts`, stable keys at the callsite.

```ts
// apps/api/src/services/i18n.ts
const TRANSLATIONS = {
  "push.wave.new.body": {
    pl: "{senderName} — nowy ping!",
    uk: "{senderName} — новий пінг!",
  },
  ...
} as const satisfies Record<string, Translations>;

export type BackendTranslationKey = keyof typeof TRANSLATIONS;
```

```ts
// callsite
void sendPushToUser(recipientId, {
  title: "Blisko",
  body: t("push.wave.new.body", recipientLocale, { senderName }),
});
```

`Translations` is `{ pl: string } & Partial<Record<LocaleCode, string>>` — PL is required as the source of truth, every other locale is optional. `as const satisfies` gives us key autocomplete + missing-key compile errors at the callsite while preserving the loose `Partial` shape inside each row.

Lookup behavior:

- Stable dot-notated keys: `"push.wave.new.body"`, `"push.group.invite.body"`, etc.
- `locale` accepts `"pl"`, `"uk"`, or any nullable value. Unknown / null / undefined → falls back to `pl`.
- A missing translation for the resolved locale (e.g. an `en` row without an EN string yet) falls back to `pl`.
- Missing key (only possible via `as` cast) → returns the key name itself so the bug surfaces visibly.
- `{placeholder}` substitution: token names match `params` keys; missing params leave the placeholder intact.

Why a centralized catalog instead of inline at the callsite or a JSON file:

- **One place to review / restyle.** When we want to soften the wave-accepted tone, all the relevant translations are next to each other — one screen, no grepping callsites.
- **Type safety.** Callsites pass `BackendTranslationKey` — typos and removed keys fail at compile time.
- **Add-a-locale story.** Add `"en"` to `LOCALE_CODES` in `@repo/shared`, fill in `en` rows in `TRANSLATIONS` one row at a time. Partial coverage ships safely thanks to PL fallback.
- **No build infrastructure.** Single TS file + the `as const satisfies` pattern. No JSON, no PO, no macro pre-compile on Bun.

Tests in `apps/api/__tests__/i18n.test.ts` cover: locale lookup, PL fallback for null / undefined / unsupported, param interpolation (string + numeric), missing-param placeholder retention, defense-in-depth fallthrough for unknown keys.

The same helper will be used for email subjects/bodies in [BLI-282](https://linear.app/blisko/issue/BLI-282) — same `TRANSLATIONS` const, new `email.*` keys.
