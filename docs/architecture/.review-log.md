## 2026-04-11 — Post-implementation review of BLI-188 (T3 promotion on tap)

**Scope:** 2-file backend + mobile fix after production diagnosis showed `analyze-pair` never firing for a real user despite repeated profile taps.

**Issues found:** 0 critical, 0 high, 2 low (both pre-existing, flagged for visibility).
**Key findings:**
- [ALIGNED] `getDetailedAnalysis` ported bidirectional block + bilateral `isComplete` gate from `getConnectionAnalysis` — switching the modal to it does not open a data-leak surface to blocked/incomplete pairs.
- [ALIGNED] Readiness checks in `ensureAnalysis` and `getDetailedAnalysis` now key off `tier === 't3'` instead of `shortSnippet IS NOT NULL`. The invariant `t3 ⟺ shortSnippet IS NOT NULL` holds today (T2 `setWhere`-guarded upsert, T3 writes both fields atomically), but tier is the explicit source of truth per BLI-184.
- [LOW, pre-existing] `ensureAnalysis` does not check blocks — a blocked pair whose modal returns `null` still triggers self-heal after 10s and enqueues `analyze-pair` work. Not a leak (returns status strings only), just wasted AI budget. Same mechanism existed pre-BLI-188 via the old `getConnectionAnalysis` path.
- [LOW, pre-existing] Soft-delete race: target user soft-deletes between map refresh and modal open → analysis procedures still load. Cross-cutting with `security.md` `security/filter-soft-deleted`; separate ticket territory.

**Docs updated in-branch:**
- `ai-matching.md` — new "Tier Invariant & Readiness Checks" section explicating the invariant and why readiness checks must use `tier`. Update marker added.
- `user-profiles.md` — `getDetailedAnalysis` added to block/isComplete consequences row and impact map row. Update marker added.

**Patterns:**
- Second regression in a week caused by conflating "row exists" with "tier ready" after BLI-185 redefined `analysisReady` semantics. Check callsites of `connection_analyses.findFirst` across the codebase for any other place that treats bare existence as T3-ready.

## 2026-04-11 — Post-implementation review of BLI-174 (AI cost tracking)

**Scope:** Full branch review after implementation, before PR creation.

**Issues found:** 2 (0 critical, 1 high, 1 medium) — both fixed in-branch before PR.
**Key findings (all addressed):**
- [HIGH] `feed` endpoint in `ai-costs.ts` router lacked a time-window filter → unbounded scan on a growing table. Fixed: added `window: WINDOW.default("7d")` parameter matching the pattern of all other breakdown endpoints.
- [MEDIUM] `evaluateStatusMatch` JSON.parse failures bypassed logging (caught by outer try/catch without a `failed` row being appended). Fixed: moved the JSON parse inside the `withAiLogging` callback with an explicit throw on malformed output, so parse failures now produce `failed` rows in the dashboard.

**Other observations (no action needed):**
- GDPR anonymization uses two separate UPDATEs for `ai_calls` — same precedent as `request_events`. Accepted as consistent.
- Vercel AI SDK v6 `usage` extraction is correct for all 9 functions (`inputTokens`/`outputTokens` for text/object, `tokens` for embeddings).
- Wrapper preserves graceful degradation across all AI functions — errors rethrow so existing fallback catch blocks still run.
- Hot-path overhead of `withAiLogging` is negligible (~2 `Date.now()` + 1 object alloc + fire-and-forget `RPUSH`) vs 100ms–2s per AI call.

**Patterns:**
- Continues the `createBatchBuffer` reuse pattern (3rd consumer: push-log, consistency-sweep, ai-calls). Pattern is battle-tested now.
- `drizzle/no-star-select` caught by `/simplify` in `feed` endpoint — consider adding Biome lint rule or pre-commit check for this.
- Admin router `resolveDisplayNames` triplication across 3 routers (push-log, queue, ai-costs) — worth extracting to a shared util in a follow-up; out of scope for this PR per scope-discipline rule.
