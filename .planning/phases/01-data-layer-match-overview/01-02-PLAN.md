This also addresses Research open question 1 by choosing option (b): persist in the DB.

```typescript
// Extract BullMQ lifecycle telemetry from the job object
const telemetry = {
  triggeredBy: job.data.triggeredBy,
  jobId: job.id ?? null,
  enqueuedAt: job.timestamp ? new Date(job.timestamp) : null,
  processedAt: job.processedOn ? new Date(job.processedOn) : null,
  processDurationMs: job.processedOn ? Math.round(performance.now() - (/* use a t0 captured at handler start */)) : null,
  waitDurationMs: job.processedOn && job.timestamp ? job.processedOn - job.timestamp : null,
  attemptsMade: job.attemptsMade ?? null,
};
```

For `processDurationMs`: The worker already captures `performance.now()` at the start of processing for logging. Use that same timing. Specifically, there are already timing variables like `const tWrite0 = performance.now()` — but for overall process duration, capture a `const t0 = performance.now()` at the very start of the analyze-pair handler (before any DB reads or AI calls) and compute `Math.round(performance.now() - t0)` right before the DB write.

Include these telemetry fields in every `db.insert(schema.connectionAnalyses).values({...})` and `.onConflictDoUpdate({ set: {...} })` call within the analyze-pair worker — there are multiple: the T3 full-analysis path (2 upserts for A->B and B->A) and the T2 fast-path (2 upserts for A->B and B->A). Each must include the telemetry fields.

Note: For the T2 fast-path upserts that use `setWhere: isNull(schema.connectionAnalyses.shortSnippet)`, include the telemetry fields in the `set` object alongside the existing fields. The telemetry should always be updated even if the T2 results don't overwrite T3 results (so the conditional update on shortSnippet is fine for the analysis fields, but the telemetry fields should ideally always be written). However, keep it simple — include them in the same conditional set. The T3 path will overwrite them anyway when it completes.
